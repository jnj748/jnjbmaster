플랫폼관리자 메뉴 구조 정비
What & Why
플랫폼관리자 모드의 사이드바가 역할(관리소장/경리·회계/시설기사/본사총괄/파트너사)마다 동일한 콘텐츠 메뉴(약관·공지·AI 자료·업무 템플릿·캠페인 알림)를 반복 노출해 XY축 중복이 발생하고 있다. 또한 약관 관리·공지 관리 페이지 안에 이미 "역할별 탭"이 있어 사이드바 반복은 사용성·정보구조 모두에서 잉여다.

추가로 disabledCategories로 카테고리 단위 비활성 로직은 존재하지만 플랫폼관리자가 직접 토글할 UI가 없고, 메뉴 정의가 permissions.ts 한 파일에 거대 if/else로 묶여 있어 메뉴 한 줄 수정이 활성/비활성·접근권한 로직 전반에 회귀를 유발하기 쉽다.

이 작업으로 (1) 사이드바 중복 제거 + 약관 관리를 "설정"으로 통합, (2) 공지/캠페인 알림 통합, (3) 유저유형별 메뉴 활성/비활성 그리드 UI 도입, (4) 향후 안전한 수정과 빠른 반영을 위해 메뉴 정의를 메뉴 블록(모듈) 단위 레지스트리로 리팩터링한다.

Done looks like
플랫폼관리자 사이드바에서 역할 그룹(관리소장/경리·회계/시설기사/본사총괄/파트너사)에는 "현황"과 그 역할 고유의 운영 메뉴(예: 시설기사 승인, 협력업체, 파트너 크레딧)만 남는다.
약관 관리·공지/캠페인 알림·AI 공통 자료·업무 템플릿은 사이드바 "설정"(또는 "콘텐츠 관리") 그룹 하나로 통합되어 단일 진입점으로만 노출된다. 각 페이지 내부의 "역할별 탭"은 그대로 유지된다.
공지와 캠페인 알림이 단일 화면("본사 알림")으로 합쳐지고, 노출될 유저유형(역할)·노출 채널(알림센터/배너/모달/푸시)·CTA가 한 폼에서 지정된다. 기존 데이터는 손실 없이 마이그레이션되어 기존 운영 중인 공지·캠페인이 그대로 보인다.
플랫폼관리자가 "설정 > 유저유형별 메뉴 활성화" 화면에서 역할 × 메뉴 블록 그리드를 보고 체크박스로 활성/비활성을 즉시 토글·저장할 수 있다. 저장하면 해당 역할의 모든 사용자의 사이드바·하단 네비·대시보드 위젯이 즉시 영향을 받고(다음 로그인 또는 라우팅 시 반영), 비활성화된 메뉴는 사이드바·하단탭·대시보드 진입 카드에서 모두 사라지며 직접 URL 접근 시에도 차단된다.
메뉴 정의가 메뉴 블록 레지스트리(블록 ID, 라벨, 아이콘, 경로, 접근 가능 역할, 카테고리 그룹, 기본 활성 여부 등을 가진 단일 자료구조)로 리팩터링되어, 새 메뉴 추가/수정 시 한 블록만 손대면 사이드바·접근권한·활성화 토글 UI가 자동 반영된다.
비활성화 상태가 서버에 저장되어 새 세션·다른 기기에서도 동일하게 적용된다(현재 disabledCategories가 사용자 단위인지 역할 단위인지 확인 후, 역할 단위 정책은 신규 저장소를 사용).
Out of scope
역할/권한 체계(RBAC) 자체의 재설계. 기존 Role enum과 ROUTES.access 정책은 유지한다.
공지·캠페인 데이터 모델의 분석/리포팅(오픈율 등) 신규 기능.
일반 사용자(비-플랫폼관리자) 사이드바의 시각 디자인 변경. 동작(비활성 메뉴 숨김)만 영향을 받는다.
다국어(i18n) 도입.
Steps
메뉴 블록 레지스트리 도입 — 플랫폼관리자뿐 아니라 모든 역할 사이드바가 참조하는 단일 메뉴 블록 정의(블록 ID, 라벨, 아이콘, 경로/쿼리, 접근 가능 역할, 그룹/섹션, 기본 활성, 모바일 하단탭 노출 여부)를 만든다. 기존 permissions.ts의 사이드바 구성 로직을 이 레지스트리 기반으로 재구현하되, 기존 노출 메뉴 집합과 1:1 호환되도록 한다(회귀 방지).

플랫폼관리자 사이드바 중복 제거 — 역할 그룹에는 "현황 + 그 역할 고유 운영 메뉴"만 남기고, 약관·공지/캠페인 통합·AI 공통 자료·업무 템플릿은 "콘텐츠 관리" 또는 "설정" 그룹 하나로 통합한다. 각 페이지는 기존의 역할 탭 UI를 그대로 사용한다(쿼리스트링 진입 동선은 유지).

공지·캠페인 알림 통합 화면 — 두 페이지를 "본사 알림" 단일 관리 화면으로 합치고, 노출 유저유형(역할 다중 선택), 채널(알림센터/배너/모달/푸시), 일정·재노출, CTA를 한 폼에서 지정한다. 기존 announcements/campaigns 데이터는 통합 모델로 안전 마이그레이션하며, 기존 라우트는 새 화면으로 리다이렉트한다.

유저유형별 메뉴 활성/비활성 그리드 UI — "설정 > 유저유형별 메뉴 활성화" 페이지를 신설한다. 행=메뉴 블록(레지스트리에서 자동 도출), 열=역할(관리소장/경리·회계/시설기사/본사총괄/파트너사). 체크 상태를 즉시 미리보기 가능하고 저장 시 서버에 영속화한다. 카테고리/그룹 단위 일괄 토글, "기본값으로 복원" 버튼을 제공한다.

활성/비활성 정책 적용 경로 확장 — 사이드바·모바일 하단탭 외에 대시보드 위젯 레지스트리, 라우팅 가드(직접 URL 접근), 알림 진입 동선도 비활성 블록을 존중하도록 한다. 비활성 시 사용자에게는 안내 페이지 또는 자동 리다이렉트로 노출한다.

저장소 모델 정리 — 현재 disabledCategories가 사용자(user) 필드로 보이는데, 본 기능은 역할 단위 정책이므로 신규 테이블/설정(role_menu_overrides 류)을 추가하고, 사용자별 오버라이드는 차후 확장을 위해 자리만 남긴다. 기존 사용자 단위 비활성 값은 마이그레이션 또는 폐기 정책을 정한 뒤 일괄 처리한다.

테스트 — 5개 역할 각각으로 로그인하여 (a) 사이드바 노출 메뉴가 그리드 설정과 일치, (b) 직접 URL 접근 차단, (c) 대시보드 카드/위젯에서도 비활성 블록이 사라지는지, (d) 통합된 본사 알림이 역할 타깃팅대로 노출되는지를 e2e로 검증한다.
